**轻量基建方案**


**核心原则**

- 先写需求，再写接口，再写测试，再写实现。
- 所有变更必须过 CI。
- 测试覆盖核心逻辑，不追求假覆盖。
- 文档跟代码一起 review。
- 设计模式只用于减少重复，不提前抽象未来。
- AI agent 只读 review，不写代码、不改文件。

**推荐最小栈**

| 领域 | 建议 |
|---|---|
| 测试 | Vitest 或 Jest |
| Coverage | V8 / Istanbul |
| Lint | ESLint |
| Format | Prettier |
| 类型检查 | TypeScript strict |
| CI | GitHub Actions |
| 文档 | MkDocs + Material |
| 设计记录 | Markdown specs + ADR |
| Review | 人工 review + 只读 review agent |

**目录治理**

建议把仓库分成几类内容：

| 区域 | 用途 |
|---|---|
| `packages/` | 代码包 |
| `docs/` | MkDocs 文档 |
| `specs/` | feature specs / requirements |
| `adr/` | 架构决策记录 |
| `tests/` 或包内测试 | 测试 |
| `.github/` | CI / PR 模板 |
| 根目录规则文件 | AI policy、贡献规范、README |

不要为了“看起来专业”拆太多层。一个人项目，目录要少、清楚、能长期坚持。

**开发流程**

每个功能按这个顺序走：

1. 写 feature spec  
   说明用户目标、输入输出、边界、错误情况、非目标。

2. 设计 interface  
   先定义模块边界、公开接口、数据结构、错误模型。

3. 写测试计划  
   列出正常路径、异常路径、边界路径。

4. 写测试  
   先覆盖核心行为。

5. 实现功能  
   只实现 spec 里的内容。

6. 更新文档  
   行为变化必须同步 docs。

7. 跑本地检查  
   格式、lint、类型、测试、coverage、构建。

8. PR / review  
   即使只有你自己，也用 PR 思维看 diff。

9. 合并前 CI 必须全绿。

**Spec 规范**

每个 feature spec 至少包含：

| 字段 | 内容 |
|---|---|
| 背景 | 为什么要做 |
| 目标 | 做成什么 |
| 非目标 | 明确不做什么 |
| 用户场景 | 谁在什么情况下使用 |
| Requirements | 必须满足的行为 |
| Interface | 暴露给外部的接口 |
| Data Model | 核心数据结构 |
| Error Model | 错误分类和处理 |
| Test Plan | 要测什么 |
| Open Questions | 未确定问题 |

原则：spec 不求长，但必须回答“要做什么”和“不做什么”。

**Top-Down Design**

每个新模块按四层设计：

1. Feature 层  
   用户要完成什么任务。

2. Requirement 层  
   系统必须满足哪些行为。

3. Architecture 层  
   哪些模块负责哪些事。

4. Interface 层  
   模块之间如何通信，输入输出是什么。

不要从函数开始设计。先从 feature 和 requirement 往下拆。

**Interface 设计原则**

接口要稳定、窄、明确：

- 输入类型明确。
- 输出类型明确。
- 错误类型明确。
- 不暴露内部实现。
- 不把第三方 API 响应直接泄漏到业务层。
- 不让调用者猜状态。
- 不让一个接口同时做太多事。

如果你在写很多重复 glue code，说明需要抽象。  
如果你还不知道第三个使用场景，不要急着抽象。

**测试策略**

测试目标：核心逻辑尽可能高覆盖，边界路径认真覆盖。

测试分层：

| 类型 | 用途 | 比例 |
|---|---|---|
| 单元测试 | 核心逻辑、纯函数、错误处理 | 最高 |
| 契约测试 | provider / adapter / 外部接口 | 中 |
| 集成测试 | 关键流程 | 少量 |
| 端到端测试 | 真正用户路径 | 很少 |

优先测：

- 数据转换。
- 参数校验。
- 错误处理。
- 边界条件。
- provider 适配。
- retry / timeout。
- 配置缺失。
- 外部响应异常。
- 日志脱敏。
- 向后兼容。

不要只测 happy path。

**Coverage 目标**

建议分阶段：

| 阶段 | 目标 |
|---|---|
| 初期 | 核心模块 80%+ |
| 稳定期 | 核心模块 90%+ |
| 长期 | 关键逻辑接近全覆盖 |

更重要的是 branch coverage。  
语句覆盖高但错误路径没测，价值不大。

规则：

- bugfix 必须带回归测试。
- 新 feature 必须带测试。
- 新 provider 必须带契约测试。
- 公共接口变更必须更新测试。
- coverage 下降必须解释。

**CI 设计**

最小 CI 必须包括：

1. 安装依赖。
2. 格式检查。
3. lint。
4. 类型检查。
5. 单元测试。
6. coverage。
7. 构建。
8. 文档构建。
9. 策略文件检查。
10. 密钥泄漏检查。

CI 原则：

- `main` 必须永远绿。
- CI 红了不能合。
- 不允许跳过测试合并。
- 不允许本地能跑、CI 不能跑。
- 不在普通 CI 里依赖真实外部 API。
- 需要真实 API 的测试放手动流程或定时流程。

**CD 策略**

一个人项目不要一开始做复杂 CD。

建议：

- 文档站：`main` 合并后自动发布。
- package 发布：手动触发。
- 生产部署：手动确认。
- 版本发布：打 tag 或 release 时触发。

早期不要自动发布 npm / Docker / 生产服务。  
CI 要硬，CD 要谨慎。

**避免 Messed Up**

重点做这些防线：

- 主分支保护。
- PR 必须过 CI。
- 不直接往 `main` 推复杂改动。
- 小 PR。
- 每个 PR 一个主题。
- 依赖更新单独 PR。
- 格式化单独 PR。
- 重构和功能分开。
- 外部 API 行为用 mock 固定。
- 关键设计写 ADR。
- 文档站每次构建验证。

**Git 规范**

提交类型保持简单：

| 类型 | 用途 |
|---|---|
| `feat` | 新功能 |
| `fix` | 修 bug |
| `test` | 测试 |
| `docs` | 文档 |
| `refactor` | 重构 |
| `ci` | CI |
| `chore` | 维护 |

提交原则：

- 一次提交一个意图。
- 不混功能和重构。
- 不混代码和大格式化。
- commit message 写清楚为什么。
- 大改动走 PR，不直接压进主分支。

**PR 规范**

每个 PR 必须说明：

- 做了什么。
- 为什么做。
- 怎么测试。
- 风险是什么。
- 是否影响接口。
- 是否更新文档。
- 是否新增/修改测试。
- 是否触碰 AI policy、密钥、网络、CI。

一个人也要写 PR 描述。  
这是给未来的你看的。

**文档站设计**

MkDocs 文档站建议包含：

| 页面 | 用途 |
|---|---|
| Overview | 项目是什么 |
| Getting Started | 怎么开始 |
| Concepts | 核心概念 |
| Architecture | 架构说明 |
| Specs | 功能设计 |
| Testing | 测试规范 |
| CI/CD | 自动化规范 |
| Git Workflow | Git 规范 |
| Review | review 规范 |
| AI Policy | AI 使用边界 |
| ADR | 决策记录 |
| Blog | 项目日志 |

博客写过程，文档写当前事实。

例如：

- “为什么选择某方案”写 blog。
- “当前架构是什么”写 docs。
- “最终决策是什么”写 ADR。

**Design Pattern 规范**

不要提前模式化整个项目。  
只对“同类功能”做模式沉淀。

适合提前设计 pattern 的地方：

- provider 适配。
- 配置加载。
- 错误处理。
- 日志。
- 重试。
- cache。
- 数据转换。
- 命令入口。
- API client。
- 测试 fixture。

每类 pattern 要写一页短文档：

| 字段 | 内容 |
|---|---|
| 问题 | 这类功能重复在哪里 |
| 适用场景 | 什么时候用 |
| 不适用场景 | 什么时候别用 |
| 接口原则 | 怎么暴露 |
| 错误处理 | 怎么失败 |
| 测试要求 | 必须测什么 |
| 示例说明 | 用文字描述结构，不放生成代码 |

判断是否该抽象：

- 出现第三次重复，再抽象。
- 抽象后代码更少、更清楚，才保留。
- 抽象不能隐藏错误。
- 抽象不能让调用方更难懂。
- 不为未来假设设计框架。

**Review Agent**

review agent 只做只读检查。

允许：

- 看 diff。
- 找 bug。
- 找测试缺口。
- 找文档缺口。
- 找架构问题。
- 找安全风险。
- 检查是否违反 AI policy。

禁止：

- 写代码。
- 改文件。
- 生成 patch。
- 自动 approve。
- 自动 commit。
- 自动修复。

review 输出固定为：

1. 阻断问题。
2. 测试缺口。
3. 文档缺口。
4. 架构风险。
5. 安全风险。
6. 非阻断建议。
7. 合并建议。

**第一阶段落地**

先做这些，不要更多：

1. README。
2. AI policy。
3. CONTRIBUTING。
4. MkDocs 文档站。
5. 最小 CI。
6. 测试框架。
7. coverage 报告。
8. PR 模板。
9. ADR 目录。
10. review checklist。

**第二阶段落地**

稳定后再加：

1. coverage 阈值。
2. 文档链接检查。
3. 依赖安全扫描。
4. release notes。
5. changelog。
6. review agent 自动评论。
7. nightly 集成测试。
8. docs 自动发布。

**不要做**

早期不要做这些：

- 复杂 monorepo 发布系统。
- 多环境 CD。
- 自动生产部署。
- 大量自定义脚手架。
- 复杂权限模型。
- 过多 CI matrix。
- 过早抽象 framework。
- 为不存在的插件生态设计接口。
- 没有真实重复就写 design pattern。

**你的最小标准**

这个项目的基建合格线：

- 每个功能先有 spec。
- 每个核心行为有测试。
- bugfix 必有回归测试。
- CI 不绿不合。
- 文档站能构建。
- 设计决策有 ADR。
- 同类功能有 pattern 文档。
- review agent 只读。
- AI 不写代码。
- `main` 始终稳定。

**一句话路线**

先用 specs 驱动设计，用 tests 锁住行为，用 CI 守住主线，用 MkDocs 沉淀知识，用少量 design pattern 控制重复。一个人项目不要重流程，但关键门禁必须硬。
